# The 4 Noble Pillars of Reproducibility

1. Files
2. Environment
3. Pipeline
4. Story

## Version Control and Project Structure

If you cannot track how your code evolved,
your results aren't reproducible.

*Git & GitHub/GitLab/Bitbucket*: Track every change to your codebase.
Never rely on filenames like script_v2_final_fixed.py.

*Standardized Project Layouts*: Use a consistent directory structure
so collaborators know where to find data, scripts and notebook outputs.
A popular framework for this is [Cookiecutter Data Science](https://cookiecutter-data-science.drivendata.org/),
also used by [Kedro](https://kedro.org/).

```
├── data/
│   ├── raw/          # NEVER touch or modify these files
│   ├── refined/      # Cleaned data generated by scripts
│   └── wrapped/      # Analysis ready data
├── notebooks/        # For exploration and prototyping
├── reports/          # For publication and sharing
├── src/              # Source code (the local Python package)
├── tests/            # Unit tests for the source code
└── uv.lock           # Dependencies
```

## Environment and Dependency Management

Code that works on your machine today might break tomorrow because a package updated.
You need to lock down your exact computational environment.

*Virtual Environments*: Use a virtual environment to isolate your project's dependencies
so they don't conflict with other projects.

*Dependency Files*: Always export a precise list of your packages
and their exact versions (e.g. a `uv.lock` file).

*Containerization*: For the gold standard of reproducibility, use Docker.
A Docker container packages your code, the operating system,
libraries and dependencies together,
ensuring it runs identically on any machine.

## Programmatic Data Pipelines

Manual data cleaning is the enemy of reproducibility.
If you copy-paste cells or manually fix a typo in a spreadsheet,
that step is lost to history.

*Scripted Pipeline Execution*: Write code (Python, R, SQL) for every step of data ingestion,
cleaning and feature engineering.
Start from the raw, unedited data every time.

*Workflow Orchestration*: For complex projects,
use workflow tools like `Snakemake` or `Kedro`.
These track data lineage (provenance)
and only re-run steps of the pipeline if the underlying data or code has actually changed.

## Literate Programming and Documentation

Your code shouldn't just run,
people need to understand why it was written that way.

*Computational Notebooks*: Use Jupyter Notebooks to blend code,
visualizations and narrative explanations together.

*Self-Contained Notebooks*: Ensure your notebooks can be run from top-to-bottom sequentially
("Restart & Run All") without throwing errors.
